# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Voxa is a unified 3D scan studio for industrial LiDAR / point clouds. It replaces the older `3d-labeler` and `industrial-point-labeler` tools. The UI is organized around four modes: **Inspect** (fast review), **Label** (see below), **Compare** (two finished labelings ‚Äî session outputs or presegs ‚Äî diffed per-point: agreement coloring, per-class IoU/precision/recall, confusion matrix), and **Edit** (non-destructive OBB slicing ‚Üí full-density PLY export).

**Label mode is unified around one principle: a tool is only a way to *select points*; everything downstream is one shared pipeline** (`select ‚Üí apply+label ‚Üí unconfirmed pointset ‚Üí confirm`). A single `activeTool` state drives a 7-tool rail in the viewport top toolbar (`frontend/src/tool-rail.jsx`, list/gating in `label-tools.js`) and one contextual tool-options panel in the left rail (`frontend/src/tool-options.jsx`):
- **Presegment** ‚Äî Ctrl/Shift-click precomputed segments (or the presegment list); a `rapid` toggle = the old fast-labeling queue (largest-first, one-keypress classify, see `frontend/src/fast-label.jsx`). The presegmentation (segment hulls/boxes + per-segment hue coloring of the cloud) is shown **only while this tool is active** ‚Äî presegmentation is a way to *select* points, not a property of the cloud. Box and Draw work on the raw RGB cloud. The gate is `isPreseg = activeTool === 'presegment'` in `mode-label.jsx` (drives `colorMode` + the `segHulls`/`segBoxes` Viewer props; the Viewer's own `segActive` recolor at `viewer.jsx` keys off whether `segHulls` is passed).
- **Box** ‚Äî draw an oriented box, transform it (G/R/Y or the panel buttons), apply ‚Üí the box vanishes and its enclosed points become a pointset. The box's OBB (`center/size/rotation`) is persisted on that pointset as a resolution-independent selection volume (still display-only, no gizmo), not a live cuboid annotation.
- **Prism** ‚Äî label any object with a non-rectangular footprint by tracing a **footprint** and extruding it. The interaction mirrors `engine/product/demo`'s Measure **Surface** tool (footprint) and **Volume** tool (height) ‚Äî see `product/demo/src/use-measure-tool.js`. **Footprint:** the **first** click snaps to the cloud surface (`firstHitUnderCursor`) and fixes a **horizontal base plane** at that height; every later corner rides that plane (ray‚à©plane) so the footprint stays **coplanar** ‚Äî a vertical prism needs a planar XZ footprint, and per-corner surface snapping produced self-intersecting (bowtie) footprints over depth-varying geometry that selected the wrong points. A dashed rubber-band previews on the same plane; **double-click or Enter** closes (min 3; Backspace drops the last, Esc clears-then-exits). This mirrors the Measure **Volume** tool (plane footprint), not the Surface tool. **Height:** on close, the footprint's camera-nearest edge defines a vertical plane; **moving the mouse aims the height live**, a **click commits** it (Backspace re-aims). **Classify:** a class hotkey / Ctrl+Enter / Apply then labels through the shared pipeline. Orbit stays enabled throughout (a drag orbits, a click places/commits), like the measure tools. Reaches the backend through the same `apply-shape` endpoint as a new `prism` shape (`{polygon:[[x,z],...], y0, height}` ‚Üí XZ point-in-polygon √ó Y-band in `shapes.py::prism_indices`). The snapped corners ‚Üí emitted shape mapping is `frontend/src/prism-geom.js::prismShapeFromCorners` (XZ projection; `y0` = min corner Y; signed-height normalization so aiming up or down both give `height>0`) ‚Äî a pure, unit-tested helper. The in-viewport preview is a **translucent-filled** footprint + extruded volume (fan-triangulated fills like the measure Surface/Volume tools) plus outline/markers ‚Äî but **no live point-highlight**; `pointsInsidePrism` in the same file is a parity-locked containment mirror of `prism_indices` kept ready for a future live point-highlight, not currently wired. The prism volume persists on the pointset (`Cuboid.prism`, `source:'prism'`) so a raw export re-rasterizes it exactly (`materialize.py::collect_volumes`/`replay_labels`), full parity with Box/Beam. Being a volume sweep, it carries `protect_instances` (confirmed = locked). Sub-mode like Beam: overlay + input in `frontend/src/prism-mode.jsx`, mounted via `PrismOptions` in `tool-options.jsx`. See `docs/superpowers/specs/2026-07-20-prism-box-labeling-tool-design.md`.
- **Draw** ‚Äî label pipes/tanks by drawing centerline paths (raw RGB cloud, walk nav, point anchors/branches, M merges; applied paths surface as pointset rows). The centerline graph persists per-session in `sessions/<id>/centerlines.json` ‚Äî see `frontend/src/draw-mode.jsx`.
- **Beam** ‚Äî label straight steel members (beams/pillars) by building a node/edge graph: Ctrl+click places joints, connecting two joints creates a square-section beam (scroll = width), Enter applies every staged/edited beam (one pointset instance per beam, re-apply keeps the id), Ctrl+Enter commits the batch into a faded read-only layer behind a ‚óå/‚óè toggle. Each beam converts to an OBB on the frontend (`beam-graph.js::obbForEdge`, Euler parity-locked to `obb_indices` via tests) and applies through the same `apply-shape` endpoint ‚Äî no beam-specific extraction code. The graph persists per-session in `sessions/<id>/structure.json` (`GET`/`PUT /api/segment/structure`, both session-pinned; geometry is not on the undo stack). See `frontend/src/beam-mode.jsx`, `beam-graph.js`, `backend/labeling/beams.py`.
- **SAM** ‚Äî label arbitrary/irregular objects with SAM (Segment Anything v3): Shift-drag a 2D box (or type a text prompt for "segment all ‚Äπpipe‚Ä∫"), a **separate GPU sidecar** renders the raw cloud from voxa's exact camera pose, runs SAM, and back-projects the mask(s) to scan-resolution point indices; the returned masks are shown in a large review modal (`frontend/src/sam-mode.jsx::SamReviewModal`). Accepting a mask does **not** classify it ‚Äî it materializes the mask's points as a candidate in a new `sam_ids` layer (`SegmentSession.sam_ids`/`sam_segments` in `backend/labeling/segment_state.py`, mirrors the existing immutable `preseg_ids` layer but is mutable and session-persistent), the same "select, don't label" split every other tool follows. Candidates show up in their own **SAM segments** list (`frontend/src/sam-segment-list.jsx::SamSegmentList`, a sibling of `PresegmentList` ‚Äî the two lists/selections are never mixed) and recolor cyan in the viewport only while the SAM tool is active (`mode-label.jsx`'s `SAM_CANDIDATE_COLOR`); Ctrl/Cmd/Shift-click a row or a cyan point to select candidates, then classify the selection through the same shared Ctrl+Enter/hotkey/class-picker pipeline every other tool uses (`mode-label.jsx::confirmSamSelection`), which is what finally produces a real `pointset` instance. Overlapping captures are last-materialize-wins (a newer accepted mask retires/shrinks an older candidate's claim on shared points, `_retire_sam_ids`); any tool that labels a point also retires its SAM candidacy. The candidate layer survives reload ‚Äî `working_sam_ids.npy` + `sam_segments.json` persist alongside the working arrays and rehydrate on session resume. The modal shows the full-size overlay image beside (not below) a mask list, each row swatched in the same palette baked into the overlay (`sam-util.js::maskColor`, mirrors `sam_sidecar/main.py::_palette`); a mask can also be picked by clicking it directly in the image ‚Äî the sidecar additionally returns `mask_index_png_b64`, a lossless per-pixel mask-index map the frontend decodes and hit-tests the click against (`sam-util.js::containPixel` accounts for `object-fit: contain` letterboxing). Click selects only that mask; Ctrl/Cmd/Shift+click adds/removes it from the selection. Projection is single-view v1 ‚Äî `/project` only keeps points visible (non-occluded) from the exact capture camera, so a mask that fully covers a chunky/curved object in 2D can still back-project to only its near-facing surface in 3D; no multi-view merge yet. The sidecar (`sam_sidecar/`, its own anaconda/CUDA process ‚Äî NOT voxa's `.venv`) holds the raw 188M-class cloud + SAM model; voxa's viewer is unchanged, used only as a pose+box source. The frontend renders nothing for SAM ‚Äî the image SAM sees is server-side (`sam_sidecar/render.py`, a numpy perspective splat), which is why the tool needs the raw source and is gated on `raw_source_available`. Backend proxy: `backend/routes/sam.py` (`POST /api/sam/capture` ‚Üí sidecar; `POST /api/sam/project` ‚Üí sidecar indices ‚Üí `SegmentSession.materialize_sam_segment(protect_instances=‚Ä¶)`, NOT `apply_reassign` ‚Äî classification only happens later, from a selection over the materialized candidates). The sidecar is reached via `VOXA_SAM_SIDECAR_URL`; scan identity is guarded by a source-fingerprint handshake (409 `{diverged:"source"}` on mismatch) so `/project` can never write indices onto the wrong cloud. See `sam_sidecar/README.md`, `docs/superpowers/specs/2026-07-12-sam-labeling-tool-design.md`, and `docs/superpowers/specs/2026-07-13-sam-segments-as-presegments-design.md`.
- **Region** ‚Äî the 7th rail tool, not for labeling but for marking scan-level *eval regions*: prism-drawn (footprint + extrude, reusing the shared draw machinery extracted to `frontend/src/prism-draw.jsx` so Region and Prism share one interaction with different commit handlers) volumes that flag where the labeling is trustworthy enough to score against. The height-commit click POSTs a draft region immediately ‚Äî no class picker, no point selection, and nothing on the undo stack, the same geometry-not-annotation treatment as Draw/Beam. Regions are **scan-level truth**, not session-scoped: they live in `<scan>/eval_regions.json` at the scan root, shared by every session on that scan, in the scan's *stored* frame ‚Äî `backend/routes/regions.py` converts to/from the runtime (recentered) frame via `regions.py::shift_prism` + the load-time `recenter_offset`. Backend (`backend/labeling/regions.py`, routes `GET/POST /api/regions`, `PATCH/DELETE /api/regions/{rid}`, `GET /api/regions/stats`) owns two server-side invariants, same philosophy as `reject_frozen_class`: the **eval-grade gate** ‚Äî flipping a region to `eval_grade` measures p50/p90 nearest-neighbour spacing over the region's FULL-RES points (`labeling.materialize.raw_sample_spacing`) and 422s below a 100-point floor, above a 10 mm p90 bar, or over the 3% `excluded_review` budget (phase 2, `REVIEW_BUDGET_FRAC`), recording `accuracy:{p50,p90,loa,measured_at}` on success ‚Äî and the **geometry lock**, which refuses geometry edits and delete on an eval-grade region until it's flipped back to draft (renames stay allowed; a name isn't geometry). The Regions tab of Label mode's right panel (`frontend/src/region-panel.jsx`, sibling of the Instances tab) lists each region with a draft/eval-grade status swatch, inline rename, unlabeled %, review % (red over budget), an eye toggle, and a gate button that surfaces a 422 refusal inline; expanding a row shows the recorded accuracy and its **majority-inside** confirmed instances (`frontend/src/region-utils.js::majorityInstances` ‚Äî an instance counts as inside a region iff >50% of its points are) computed frontend-side over the backend's raw per-region `{inside,total}` counts, because confirmed-status lives in `instances_gt.json`, not the working arrays ‚Äî the same split as `protect_instances`. Overlay visibility: all regions render while the Region tool is active, plus the per-region eye-toggled set under any other tool (UI-local, not persisted). See `docs/superpowers/specs/2026-07-21-eval-regions-design.md`.
- **Cut selection** ‚Äî not an 8th rail tool but a right-click "Edit selection‚Ä¶" affordance on the three list surfaces (`PresegmentList`, `SamSegmentList`, the Instances panel), for narrowing an existing selection instead of discarding it: cut a sub-region out of a multi-selected presegment/SAM-candidate set, or out of a single unconfirmed instance, and the removed points get re-partitioned into new selections/instances rather than deleted. The trigger is list-row-only, never the viewport ‚Äî right-click there is already fully claimed by `attachOrbit`/`attachWalk`'s pan-drag, so there's no free gesture to repurpose without touching camera-control code. `cutEligibility` (`frontend/src/cut-eligibility.js`) gates the menu item: preseg/SAM lists need `selectionSize > 0`, the Instances panel needs a single selected + unconfirmed instance (confirmed = locked, same rule as everywhere else). Choosing "Edit selection‚Ä¶" calls `mode-label.jsx::openCutModal`, which mounts `CutModal` (`frontend/src/cut-mode.jsx`) ‚Äî a second, isolated `Viewer` instance fed only the selected sources' points (`buildCutCloud`, tagging each filtered point with its originating `{kind, segId}`), with its own Box-tool draw/transform state (G/R/Y, its own `selBox`) rather than a shared `box-tool.js` module ‚Äî that extraction was tried and found unnecessary; `Viewer` itself is already the reusable unit. Confirming a drawn box round-trips through `POST /api/segment/cut-shape` (`backend/routes/segment.py::_cut_shape_core`), which resolves the box to FULL-RES point indices the same way `apply-shape` does, then partitions that index set per source against that source's own full-res membership (`preseg_ids`/`sam_ids`/`instance_ids`) ‚Äî sources never merge, each non-empty partition materializes independently. A preseg/SAM partition goes through `SegmentSession.materialize_sam_segment(indices, source=...)` (the `source` param, now required, is what generalizes the SAM candidate layer into a shared cut-target layer): `source='sam'` results land back in the SAM segments list, `source='preseg'` results show up in `PresegmentList` as a "Cut selections" sub-section (`Cut #{id}` rows, selected via `samSelection`, not `segState.selection`, since they physically live in `sam_ids` regardless of the display tag). An instance partition instead auto-inherits the source instance's class (no class picker ‚Äî `mode-label.jsx::onCutConfirmedHandler` reads it off the right-clicked Instances-panel row) and reassigns through the normal `apply_reassign`/`protect_instances` path, single-instance-only (no multi-select on the Instances panel today). Known, disclosed gaps: `source:'preseg'` cut candidates get list-row selection only ‚Äî no distinct 3D viewport recoloring or point-click selection while the Presegment tool is active; Box is the only cut shape (no lasso/freeform); no affordance to discard an unwanted cut candidate; mixing presegment/SAM and instance sources in one cut is out of scope (instance cuts are single-instance only); and the modal has no live in-viewport containment preview ‚Äî the actual partition is always resolved server-side on confirm, never previewed client-side before that. See `docs/superpowers/specs/2026-07-14-cut-selection-tool-design.md`.

- **Outlier filtering** ‚Äî like Cut selection, not a rail tool but two affordances for stripping spatially-isolated stray points (sensor speckle; SAM back-projection edge-strays). One shared **statistical outlier removal** pure fn (`backend/labeling/outliers.py::statistical_outlier_indices(positions, subset_idx, k=16, std_ratio)`): build a `cKDTree` over `positions[subset_idx]`, flag points whose mean distance to their `k` nearest neighbours exceeds `mean + std_ratio¬∑std`. The only difference between the two features is the *population* (`subset_idx`). **(C) Global denoise** ‚Äî a "‚úß Detect outliers" button + aggressiveness slider (`std_ratio`, 1‚Äì3) in Label mode's left rail (next to Export; annotated + active-session only). `POST /api/segment/denoise` runs SOR over the whole cloud (reusing the session's cached `_ensure_tree()`, so a strength re-run doesn't rebuild a multi-million-pt tree) and materializes the outliers as **one unconfirmed class-less review blob** (phase 2: `excluded_review` category + a fresh instance id ‚Äî it used to write class `unknown`/id 6, which the eval-labeling spec bans in new GT). Review-first: eyeball the caught points, tweak the slider, re-run. Re-run replacement is **backend-owned** ‚Äî the frontend passes the tracked instance id as `replace_inst` and `/denoise` erases that instance's points to unlabeled *before* recomputing (the frontend's `deleteInstance` only drops the metadata row, never the working-array labels, so a frontend-only delete would orphan the old review points). Protects confirmed instances (`protect_instances`). **(B) Per-selection "Remove outliers"** ‚Äî a right-click item (gated by `frontend/src/outlier-eligibility.js::removeOutliersEligibility`) on `SamSegmentList` rows (exactly one candidate selected ‚Äî the endpoint targets a single `{source,id}`, unlike cut's multi-select) and unconfirmed Instances-panel rows (confirmed = locked). `POST /api/segment/denoise-selection` runs SOR scoped to that selection's own membership (so an edge-stray far from the selection's core is flagged even though it's near real geometry cloud-wide) and strips the outliers **back to unlabeled** ‚Äî SAM candidate via `SegmentSession.remove_sam_points` (`_retire_sam_ids` + persist), instance via `apply_reassign` erase. Presegs are out of scope (immutable `preseg_ids` layer ‚Äî noisy presegs are served by global denoise or the Cut tool). B's frontend threads `onRemoveOutliers` through `tool-options.jsx`'s `SamOptions` exactly like `onEditSelection`, and reuses the now-exported `segment-state.js::retireSamIdsForIndices` for the local SAM shrink. See `docs/superpowers/specs/2026-07-20-outlier-detection-filtering-design.md`.
- **Point categories + components** ‚Äî the third labeling axis (eval-labeling phase 2, spec `docs/superpowers/specs/2026-07-22-point-categories-components-design.md`): class says *what an object is*, instance says *which object*, and the per-point **category** says *how much we trust the annotation* ‚Äî `artifact` (no real surface), `transient` (person / self-mobile), `excluded_review` (real, identity uncommittable). Confirmed-thing/stuff is never stored; it is derived from class+instance. The vocabulary has one home per side (`backend/labeling/categories.py`, `frontend/src/point-categories.js`, pinned in sync by a test). `SegmentSession.categories` is an int8 full-res array written by one op, `apply_category` (`POST /api/segment/apply` op `set_category`; `apply-shape`'s `target_category` for Box/Prism), which rides the same undo stack and `protect_instances` lock as every other apply. A mark *resolves* the other two axes rather than layering over them: artifact/transient/none erase to `(-1,-1)`, `excluded_review` erases the class and allocates one fresh instance id ‚Äî a **review blob** (the spec's "instance-shaped blobs, never a point spray"), which surfaces as an Instances-panel row with `cls: null`, grey, labelled "Review". Symmetrically, **every other op clears the category of points it writes** (labeling a point as a real object retracts its non-object mark) ‚Äî the same unconditional treatment as `_retire_sam_ids`. Marking is reached from the shared class picker's **Non-object** row (`a`/`t`/`r` + `x` to clear) ‚Äî so Presegment, SAM, Box and Prism all get it; Draw/Beam stay class-only. Marked points recolor in the viewport through the existing selection-overlay buffer (they carry class -1 and would otherwise be invisible), and the left rail shows a per-category counter. Persistence: `sessions/<id>/working_categories.npy` (autosave; absent = a pre-phase-2 session, i.e. all `none`) and, on explicit save, `output/gt_point_category.npy` plus the derived `output/gt_point_component_ids.npy` and metadata (`categories` histogram, `review_blobs`, `component_link_radius_m`). Review blobs are class-less, so the existing save-path strip drops them from `gt_segment_ids` ‚Äî SCHEMA invariant 3 keeps passing and the blob ids live in the metadata (`review_blobs`), which phase 3's eval-invariant 3 now checks directly (see below) ‚Äî voxa kept its own `gt_*` file names rather than adopting the upstream spec's illustrative ones. **Components** (`backend/labeling/components.py::component_ids`) are the fragment index *within* an instance (int16, -1 outside), derived at save time by voxel-grid 26-connectivity at a 5 cm link radius ‚Äî an occlusion-split pipe is one instance in two components, so instance metrics can score object-level and fragment-level separately. Nothing is labeled by hand: labelers link fragments by *merging instances*.
- **Eval invariants + manifest** (eval-labeling phase 3, spec `docs/superpowers/specs/2026-07-22-eval-invariants-manifest-design.md`) ‚Äî the 9 GT-semantic loader invariants from the upstream spec, implemented as pure functions in the shared `scan_schema` package (`scan_schema.eval_invariants`, distinct from and separately numbered from the pre-existing SCHEMA invariants 1-6 in `scan_schema.invariants`) plus a scan-level manifest generator (`scan_schema.manifest::build_manifest` ‚Äî class/category histograms, in-region fraction, thresholds, provenance). Voxa's save-time gate (`backend/labeling/segment_io.py::save_labels`, via `_check_eval_invariants`) runs all 9 as hard failures before writing, surfaced as `EvalInvariantError` ‚Üí HTTP 422 (`backend/routes/segment.py::segment_save`, vs. 400 for the older SCHEMA invariants); each check is gated on its own optional input (`frozen_ids`, `instances_doc`, `eval_regions`, `prior_segment_metadata`) being supplied, so pre-existing callers that omit them are unaffected. The same functions wire into `scan_schema.validate_archive` for load-time archive auditing (`scan_schema validate --frozen-ids 0,3,4,5,6,13`) ‚Äî invariants 1/2 (category exhaustiveness, review budget) and 6/7 (manifest drift, id lineage) are honestly **not** wireable at that layer (1/2 need real per-point region masks rasterized from prism geometry + loaded positions, which only voxa's save-time gate has; 6/7 need a prior-save snapshot, which a single on-disk audit pass doesn't have) and are documented as such rather than faked. `scripts/migrate_eval_invariants.py` brought the three precious pre-phase-2 scans (`water_pump_navvis_3m`, `water_treatment_navvis`, `smart_ais_navvis`) into compliance: converting legacy frozen-class-6 points to the phase-2 review-blob representation, and ‚Äî a data-integrity issue discovered during rollout, unrelated to class-6 ‚Äî stripping "orphaned" instances (array-level labels with no `instances_gt.json` row, traced to raw never-confirmed presegment suggestions baked into old sessions by pre-instances_gt.json behavior) back to unlabeled. `backend/requirements.txt` pins `scan-schema` to a commit SHA (see `docs/scan-schema.md`) rather than tracking `@main`, so this and future scan_schema changes ship to voxa only via an explicit dependency-bump commit.
- **Fit box to selection** ‚Äî like Cut, a right-click "Fit box to selection‚Ä¶" affordance on the same three list surfaces (`PresegmentList` incl. its cut-candidate menu, `SamSegmentList`, the Instances panel), gated by `fitEligibility` (`frontend/src/fit-eligibility.js`; preseg/SAM need `selectionSize > 0`, the Instances panel needs a single selected instance ‚Äî a *confirmed* instance IS eligible here, unlike Cut, since fitting only reads the source points). It captures the selection as `[{kind, segId}]` and calls `POST /api/segment/fit-box` (`backend/routes/segment.py::_fit_box_core` ‚Üí `backend/labeling/fit_box.py::fit_gravity_obb`), which resolves the selection to FULL-RES points (same membership rules as `_cut_shape_core`) and fits a **gravity-aligned (yaw-only) OBB** ‚Äî Y locked to world-up, footprint = the exact minimum-area rectangle in X/Z via rotating calipers over the convex hull, so `rotation:[0,ry,0]` (no Euler-parity hazard). `mode-label.jsx::fitBoxToSelection` stages the returned OBB as the Box tool's `selBox` (the full `{id:LABEL_SEL_BOX_ID,...}` shape, or it won't render/transform) and switches to the Box tool. It **labels nothing** ‚Äî from there it is an ordinary editable Box selection (G/R/Y, then apply through the normal `apply-shape` pipeline, which sweeps every full-res point in the volume and honors `protect_instances`). Because apply carries `protect_instances`, fitting to a *confirmed* instance labels only the surrounding unconfirmed points into a new instance ‚Äî it cannot extend the confirmed one (un-confirm first). `onFitBox` threads through `tool-options.jsx` to the lists, mirroring `onEditSelection`. See `docs/superpowers/specs/2026-07-20-fit-box-to-selection-design.md`.

Every apply produces a `kind:'pointset'` instance (the geometric `cuboid` type is retired for new labels ‚Äî legacy cuboids render display-only, no gizmo). Apply = Ctrl+Enter ‚Üí class picker, or a class hotkey directly. A per-tool **auto-confirm on apply** toggle is the only deviation from unconfirmed-first (rapid preseg defaults on). Geometric tools (Box, Prism, Draw, Beam) reach the backend through **one generic endpoint** `POST /api/segment/apply-shape` (`backend/routes/segment.py` + `backend/labeling/shapes.py`): it resolves a shape (`obb` | `tube` | `prism`; Beam sends per-beam `obb`s, so no `beam` shape type exists) to the FULL-RES points inside it via `shape_indices`, then `apply_reassign`s them ‚Äî this is why Box/Draw need the backend (the rendered cloud is subsampled but labels are full-res). `centerline-apply` is a thin `tube`-shaped alias of `apply-shape`.

**Confirmed = locked.** A volume apply is last-write-wins over *unconfirmed/unlabeled* points, but it must **never overwrite points that belong to a confirmed instance** ‚Äî the guard against an overextended box silently eating already-labeled geometry (e.g. the floor). The frontend owns confirmed-status (it lives in `instances_gt.json`, not the working arrays), so `apply-shape`/`centerline-apply` carry `protect_instances` = the confirmed instances' `segId`s (`mode-label.jsx::protectedSegIds`, threaded through `tool-options.jsx` into Draw/Beam); `apply_reassign(protect_instances=‚Ä¶)` drops those points before the reassign and returns `n_protected`. `target_inst` and `-1` are never protected (a re-applied box may rewrite itself; unlabeled points are always labelable). To re-label a confirmed instance, un-confirm it first. Presegment click-select (the direct `reassign` path, not `apply-shape`) is deliberately **not** protected ‚Äî it is a single-segment pick, not a volume sweep.

## Commands

```bash
npm install                         # first run only
npm run dev                         # FastAPI (backend) + Vite (frontend) via concurrently
npm run build                       # build frontend ‚Üí ./dist
./scripts/run.sh                    # backend only (after build, single-port mode)
./scripts/import_scene.sh <name> <path-to-ply-or-glb>

npm test                            # frontend (vitest) + backend (pytest)
npm run test:frontend               # vitest only
npm run test:backend                # pytest only (auto-installs dev deps into .venv)
.venv/bin/pytest backend/tests/test_smoke.py::test_health   # single backend test
npx vitest run src/api.test.js      # single frontend test (run from ./frontend or with --root)
```

`npm run dev` opens the dev UI at `http://127.0.0.1:5173` and Vite proxies `/api/*` to the backend at `http://127.0.0.1:8765`. After `npm run build`, `./scripts/run.sh` alone serves both `/api/*` and the static frontend on port 8765.

`scripts/run.sh` auto-creates `voxa/.venv` and installs `backend/requirements.txt` on first run. `scripts/test.sh` does the same for `requirements-dev.txt`.

### Useful env vars

- `VOXA_PORT`, `VOXA_HOST` ‚Äî backend bind (defaults `127.0.0.1:8765`)
- `VOXA_DATA_DIR` ‚Äî overrides `./data` for scenes + annotations
- `VOXA_LIDAR_ROOT` ‚Äî root of the canonical lidar archive (default `/home/hendrik/coding/engine/data/lidar`). Adds `annotated/` scenes to the picker; set to a missing path to disable.
- `VOXA_CONFIG` ‚Äî path to a `classes.yaml` (default `config/classes.yaml`)
- `VOXA_MAX_POINTS` ‚Äî server-side subsample cap (default `1000000`)
- `VOXA_RELOAD=1` ‚Äî enable uvicorn `--reload` (off by default to avoid hitting the system inotify limit)
- `VOXA_BACKEND`, `VOXA_FRONTEND_PORT` ‚Äî Vite dev proxy target / port
- `VOXA_SAM_SIDECAR_URL` ‚Äî base URL of the SAM sidecar (e.g. `http://127.0.0.1:8011`). Unset ‚Üí the SAM tool is disabled (`/api/sam/*` returns 503). The sidecar is a separate GPU/anaconda process (`sam_sidecar/`, see its README), not part of `npm run dev`.

## Architecture

**Backend** (`backend/`, FastAPI, served by `uvicorn main:app`):
- `main.py` ‚Äî assembles the FastAPI app and registers routers from `backend/routes/*.py`. HTTP endpoints live in `routes/{load,segment,sessions,compare,export,meta}.py`; Pydantic schemas in `app/schemas.py`; in-memory `_state` + helpers (`_recenter`, `_resume_session`, ‚Ä¶) in `app/core.py`. Compare scoring is per-point (class agreement, per-class IoU/P/R, confusion matrix); there is no longer any cuboid-based IoU diff.
- `scene_registry.py` ‚Äî multi-root scene discovery. Returns `SceneSource` for each tier (`legacy` / `annotated`). Scene IDs are tier-prefixed (`annotated/munich_water_pump`); bare legacy names still resolve.
- `lidar_io.py` ‚Äî `load_annotated` (SCHEMA-conformant v2 scans; reads working arrays from `sessions/<id>/`) and `load_laz` (chunked, stride-sampled via `laspy[lazrs]`). Auto-recenter for float32 stability lives in `app/core.py::_recenter`.
- `point_cloud.py`, `supervoxels.py`, `clustering.py`, `fitting.py` ‚Äî carried over from the old `3d-labeler`. Only the PLY/GLB loader and a small `auto-fit` (snap a cuboid to points inside an AABB) are wired into the current frontend; the supervoxel / cluster / RANSAC modules are present for future use but not exposed via routes.
- Static frontend is mounted at `/` **only if `dist/` exists**, so `/api/*` always wins. In dev mode (`dist/` absent) the FastAPI process is API-only and Vite owns the UI.

**Frontend** (`frontend/src/`, Vite + React 18 + Three.js, no TypeScript):
- `main.jsx` mounts `<App>` plus `<Agentation>` (in-app feedback toolbar from the `agentation` package).
- `App.jsx` is the shell: owns `scenes / activeScene / cloud / classes / gtInstances / predInstances` state, fetches config + scenes once, then re-loads cloud + GT + predictions whenever `activeScene` changes. It dispatches to one of three mode components and handles `‚åòS/Ctrl+S` save. The `ScenePicker` surfaces **annotated scenes plus the `legacy/test_scene` sandbox** ‚Äî all other legacy scenes are filtered out of the picker UI (the backend still discovers/resolves them for internal guards). When no scene is selected the default falls back to `legacy/test_scene`, then the first scene.
- `mode-inspect.jsx`, `mode-label.jsx`, `mode-compare.jsx` ‚Äî feature surfaces. They share `viewer.jsx` (the Three.js viewport with `attachOrbit` and `attachWalk` camera schemes; `navMode` is lifted to `App` so it persists across mode switches) and the small atoms in `viewport-atoms.jsx`.
- `api.js` is the only place that talks to `/api/*`. Point cloud `positions`/`colors` come over the wire as base64-encoded `Float32Array` payloads ‚Äî decode via `b64ToFloat32`.
- `tweaks-panel.jsx` is a generic tweak-host (theme/mode controls). It implements an external "edit mode" host protocol via `postMessage` (`__activate_edit_mode` / `__edit_mode_set_keys` / etc.) ‚Äî do not delete those listeners without checking the Agentation integration.

**Data layout** (gitignored):
```
data/
‚îú‚îÄ‚îÄ scenes/<name>/source.{ply,glb}              # input
‚îî‚îÄ‚îÄ annotations/<name>/{ground_truth,predictions}.json
```
Both annotation files share one schema (`AnnotationDoc` in `app/schemas.py`); a model's output dropped at `predictions.json` is enough to power Compare mode.

**Annotated scan layout** (under `$VOXA_LIDAR_ROOT/annotated/<name>/`):
```
<name>/
‚îú‚îÄ‚îÄ meta.json              schema_version "2.x" (grandfathered) or "3.0"
‚îú‚îÄ‚îÄ source/scan.ply        point cloud (required)
‚îú‚îÄ‚îÄ prelabel/<preseg_id>/  each pipeline result: instance_ids.npy, segment_summary.json, meta.json
‚îî‚îÄ‚îÄ sessions/<session_id>/ each labeling session: session.json, working_*.npy, output/gt_*, history/
```
No `labels/`, `session/`, or `annotation_history/` ‚Äî those are absorbed into `sessions/`.
The schema (layout, meta contract, fingerprints, invariants, validation, lineage, durable writes) is
defined once in the shared **`scan_schema`** package (`tools/scan-schema`, declared as a VCS dependency
in `backend/requirements.txt`) ‚Äî voxa imports it rather than re-implementing it. See `docs/scan-schema.md`
for the voxa-facing subset and the cross-tool `lidar/SCHEMA.md` for the full contract.

**Class config**: `config/classes.yaml` ‚Äî the v7 primitive vocabulary (eval-labeling phase 0, `docs/superpowers/specs/2026-07-21-primitive-vocabulary-phase0-design.md`): 34 assignable primitives in 8 chord groups + 6 frozen legacy classes, mirroring `engine/data/lidar/classes.json` v7 (append-only; `backend/tests/test_class_config.py` pins the two in sync). Each entry has `id` (canonical numeric id), `label`, `color` (hex or `[r,g,b]` floats), `key` (the **within-group** chord second-stroke), `group` (chord group; `legacy` = no chord), and optionally `frozen: true` (display-only legacy ‚Äî renders, counts, toggles, but can never be newly assigned; backend guard `app/core.py::reject_frozen_class` 422s every assign path incl. merge/cut-inherit, with the denoise id-6 write deliberately exempt until phase 2). Class hotkeys are **two-stroke chords**: group key `1`‚Äì`8`, then the member key (`frontend/src/class-chords.js` is the pure state machine; consumed by mode-label's global handler, `ClassPickerModal`, and rapid-mode `FastLabelKeys`). Instances carry optional phase-0 metadata (`flags[]` ‚äÜ {boundary_uncertain, incomplete}, `subtype`, `insulated`, `note`) edited in the Instances-panel inspector, inert pass-through on `Cuboid`. Restart the backend or re-fetch `/api/config` to pick up edits.

## Conventions and gotchas

- **EDITMODE markers**: `App.jsx` defines tweak defaults inside `/*EDITMODE-BEGIN*/ ... /*EDITMODE-END*/`. The Agentation/tweaks tooling rewrites this region, so preserve the markers verbatim when editing defaults.
- **Vite uses polling watchers** (`vite.config.js`) on purpose ‚Äî to avoid blowing the system-wide inotify limit when many other projects are open. Keep it.
- **Backend has no autoreload by default** ‚Äî after editing Python files, kill and restart `npm run dev` (or set `VOXA_RELOAD=1`). This matches the broader "restart server after code changes" rule from the global config.
- **Single in-memory point cloud**: `_state` in `app/core.py` holds one cloud at a time. Endpoints that need the points (e.g. `auto-fit`) implicitly depend on a prior `/api/load`. If you add multi-scene workflows, lift this state out first.
- **Multi-session labeling model**: each annotated scan holds N labeling sessions under `sessions/<session_id>/`. The backend keeps ONE active `SegmentSession` at a time. Label mode (annotated tier) shows a session picker: list, create (name + preseg or blank), rename, delete, switch. Opening a scan auto-resumes the last-worked session (max `saved_at`). Save (Ctrl+S) writes to the active session's `output/` dir; downstream consumers enumerate `sessions/*/output/` ‚Äî there is no single canonical GT slot. The cuboid/pointset instance doc (right Instances panel) is also session-scoped: `sessions/<id>/instances_gt.json` via `?session_id=` on `/api/annotations/*` (the scene-global `data/annotations/` path is legacy-tier only).
- **Session pinning + 409**: each session is pinned at creation to a preseg fingerprint (`prelabel/<id>/meta.json::fingerprint`) and a source fingerprint. On resume, voxa string-compares both pins. Any mismatch returns HTTP 409 `{detail, diverged: "preseg"|"source"}` and renders as a blocking banner in the UI. Re-registering the preseg (via `register_preseg()`) is the only supported way to update a preseg.
- **Per-point labels are read-only in Inspect** ‚Äî annotated scans load working arrays from the active session (`sessions/<id>/working_*.npy`), surfaced through `LoadResponse.class_ids` / `instance_ids` / `class_palette`. Inspect's "Color by Class / Instance" pills consume them. Editing per-point segments is Label mode only.
- **Scan directory schema** is what voxa expects on disk for an annotated scan (source.ply, prelabel/<preseg_id>/, sessions/<session_id>/, renders/, sam3/, ...). See `docs/scan-schema.md`. Discovery delegates to `scan_schema.metadata.check_meta`: 2.x scans are grandfathered (missing frame/derivation are warnings) and 3.0 scans are fully validated ‚Äî both are discovered. Legacy v1.3 scans (with `labels/`, `session/`, `annotation_history/`) are not discovered ‚Äî run `scripts/migrate_scan_v2.py` first. Promote a 2.x scan to v3.0 lineage with `scripts/scan/promote_to_v3.py` (registers its raw root + writes the v3.0 nested derivation via `scan_schema.Registry.set_derivation`).
- **Auto-recenter on load** ‚Äî `_recenter` in `app/core.py` subtracts the mean centroid (`points.mean(axis=0)`) when any coord exceeds 1e3 (LAS UTM scenes). The offset is in `LoadResponse.recenter_offset`. Cuboid endpoints operate in the recentered frame.
- **Cuboids are retired for new labels** ‚Äî Label mode no longer creates `kind:'cuboid'` instances (every apply produces a `pointset`). Legacy cuboid instances in already-saved sessions still render and are selectable/confirmable/deletable but display-only (no gizmo). The `Cuboid` schema + its `rotation` field remain for those legacy instances; `rotation` has no effect on Compare (which is per-point; `_iou_aabb` was removed).
- **Box pointsets now persist their OBB (`center/size/rotation`)** as a resolution-independent *selection volume* ‚Äî narrow reversal of the above, for `source=='box'` and `source=='beam'` only. It stays `kind:'pointset'`, so it is still display-only (all gizmo/cuboid-edge/auto-fit paths gate on `kind !== 'pointset'`); the OBB is inert data a future export rasterizes at any density. Every instance also carries a monotonic apply-order `seq`, stamped/backfilled on save/load in `backend/routes/compare.py::_ensure_seq`. See `docs/superpowers/specs/2026-07-10-resolution-independent-labels-design.md`.
- **`backend/labeling/materialize.py`** is the density-agnostic label materializer that consumes the above: `materialize(ctx, resolution)` produces per-point `(positions, colors, class_ids, instance_ids)` at any density ‚Äî regime A (`‚â§ scan.ply`: exact index-transfer) or regime B (denser/raw: NN + volumetric max-`seq` replay, box/tube/beam boundaries exact (a beam's swept box is a persisted OBB, `source:'beam'`), preseg NN-approximate). The raw cloud resolves via `derivation‚Üísources.json` (`raw_source_available` on `LoadResponse`). Its consumer is the **export wizard** (Phase B, shipped ‚Äî see below): `POST /api/labels/export` in `backend/routes/export.py`. For a real raw (~156M) cloud, stream `materialize_raw(...)`'s generator ‚Äî do NOT call `materialize(kind="raw")` (it assembles in memory).
- **Export wizard** (`frontend/src/export-wizard.jsx`, opened from an "‚¨á Export‚Ä¶" button in Label mode's left rail; disabled without an active session) turns the active session's labels into a downloadable zip (labeled `scan_labeled.ply` + `manifest.json`) at a chosen resolution (`scan`/`subsample`/`raw`) with confirmed-only / include-exclude / class-remap filters. Backend: `POST /api/labels/export` (materialize ‚Üí pure filter/remap pipeline in `backend/labeling/export_pipeline.py` ‚Üí binary PLY + manifest ‚Üí zip to a temp dir, `FileResponse` with `try/finally` + `BackgroundTask` cleanup); `GET /api/labels/accuracy` feeds the Review step's boundary-uncertainty line (USIBD LOA band from `labeling.materialize.loa_band` + raw p50/p90 nearest-neighbor sample spacing; the exact computation is embedded in `manifest.json::accuracy.method`). Read-only ‚Äî never mutates stored labels. Pure-fn helpers: `frontend/src/export-wizard-util.js` (`remapToTaxonomy` mirrors the backend `build_taxonomy`). Raw is disabled unless `raw_source_available`. See `docs/superpowers/specs/2026-07-10-export-wizard-design.md`.
- **Coordinate system**: Three.js Y-up. Cuboid `center`/`size` are in scene units; `rotation` is `[rx, ry, rz]` Euler XYZ in radians. Every OBB containment test (backend `scenes/reproject.py::euler_xyz_matrix` ‚Üí `shapes.py`/`_obb_mask`, frontend `mode-edit.jsx::pointsInsideOBB`, `mode-label.jsx` preview) composes it as **Rx¬∑Ry¬∑Rz** ‚Äî the matrix Three.js `Euler(..., 'XYZ')` renders with. Don't hand-roll a new one; single-axis tests can't catch a reversed composition.
- **Undo/instance-doc invariants**: fresh instance ids are session-monotonic (`SegmentSession._next_fresh_inst` ‚Äî an undone apply's id is never re-issued), and Ctrl+Z/Ctrl+Shift+Z reconcile the Instances panel via `segment-state.js::reconcilePointsetRows` (an undone apply's pointset row goes dormant so its OBB volume doesn't replay into raw exports; redo revives it). `instances_gt.json` is written atomically (`put_annotation`) and `getAnnotation` throws on failure rather than returning an empty doc ‚Äî never reintroduce a silent empty-doc fallback; the autosave would persist it over the session's real instance metadata.

## Tests

Most tests are pure-function; a handful of component tests (`context-menu.test.jsx`, `sam-segment-list.jsdom.test.jsx`, `segment-tools.jsdom.test.jsx`) opt into `jsdom` + `@testing-library/react` via a `@vitest-environment jsdom` pragma.
